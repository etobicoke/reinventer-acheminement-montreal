Montréal open data — field notes, dead ends, and traps

Companion to proposal.html · 2026-07-15 · the other half of the work: what cost the most to learn and appears nowhere in the pitch, so the next person doesn't rediscover it. Everything here is re-derivable via shadow/montreal-challenge-verify.mjs.
Why this document exists The proposal necessarily leads with what works; this is the complete record — including what didn't, so the next person (including us in September) doesn't walk into the same walls. Read this before touching Montréal's open data again.

1Fields that do not mean what they are named

This is the whole document in one table. Every row cost us at least one wrong conclusion.

2Dead ends we walked into

The 51% that wasn't

Comparing ARRONDISSEMENT to ARRONDISSEMENT_GEO with naive string equality reports a 50.9% mismatch. It is almost entirely formatting: Mercier - Hochelaga-Maisonneuve vs Mercier-Hochelaga-Maisonneuve. Normalised, the true divergence is 0.69% — and even that is not an error rate, because the two fields mean different things (who fixes it vs where it is).

LessonA shocking number from a string comparison is a string bug until proven otherwise.

The circular zero

Filtering pothole requests to villes liées on ARRONDISSEMENT returns 0 — which looks like a finding ("citizens never misreport across the boundary") and is actually a restatement of the column's definition. The right column, ARRONDISSEMENT_GEO, returns 311: Mont-Royal 169, Westmount 91, and more. Westmount's most common 311 category is Nid-de-poule.

LessonThe correct number supported the argument better than the wrong one. Convenient findings deserve more scrutiny, not less.

The owner column that isn't

We found PROPRIETAIRE_REF, read its value dictionary — which lists exactly the third parties we hoped for — and concluded "the jurisdiction join is a column." We never checked whether the values were populated. They are not.

LessonA column's existence and its permitted-value list are not evidence about its contents. Check the distribution, never the schema.

3The best find: the obfuscation rule

Not in the CKAN metadata — it is rendered on the dataset page only, section "Production des données":

« Pour respecter la confidentialité des requérants, cette position géographique est ensuite obfusquée: les requêtes associées à des adresses ou installations particulières sont relocalisées au milieu du tronçon le plus près ayant une longueur de plus de 45 mètres. »
Why it matters twice Architecturally: the City's privacy team independently chose the tronçon as the atomic safe unit of location — the same unit jurisdiction attaches to. Privacy and routing want the same primitive. Forensically: it explains every coordinate anomaly — and we mis-diagnosed all of them before finding it.
ObservationOur wrong readThe truth
19,024 distinct coords for 56,296 potholesheavy obfuscation / grid snapping~19k tronçon midpoints; many potholes per segment
148 reports on one exact coordinatea suspicious hotspot148 potholes on one road segment over 4 years
Two hot points ~2 m apartduplicate geocodesthe same segment's midpoint recomputed after a géobase version bump — which is why geobase-gestion-troncon exists

Consequence for any eval: the published coordinates are segment midpoints. You cannot validate metre-precision asset resolution against 311 history. You can validate segment- and territory-level routing — which is the granularity jurisdiction lives at anyway. The 45 m threshold also skips short segments, so any accuracy measured through it is a floor.

4Confounds we could not eliminate

  • Internal work orders are indistinguishable from citizen requests. The City: « Certains arrondissements utilisent également la plateforme pour assigner des tâches internes… Le système ne permet pas de distinguer ». These are created by the unit that performs them, which inflates the geometry rule in our favour. Unsizable.
  • Possible tautology. If UNITE_RESP_PARENT is auto-assigned from the coordinate, "geometry predicts the unit" partly restates that rule. The shared-service exceptions prove it is not purely geometric, but we could not rule this out.
  • Ground truth is the endpoint only. Reroutes are invisible; we reproduce where a request ended, not the decisions along the way.

Gotchas, all hit for real:

  • /download/ paths 302 to Google Storage; in a sandboxed shell the cross-host redirect returns RBAC: access denied (19 bytes). The API works fine; only bulk file downloads are affected.
  • The dataset page is not curl-able (times out). The methodology text needs a rendering fetch — it is not in the API.
  • COUNT(*) FILTER (WHERE …) silently returns nothing on some datastore queries. Use SUM(CASE WHEN … THEN 1 ELSE 0 END) or separate queries.
  • The column is PROPRIETAIRE_REF, not PROPRIETAIRE. Dictionary attribute-group names are the literal column names.
  • Paginate grouped queries — the datastore caps rows per response.

6What is still unverified

Do not present any of these as fact without checking first.

  • Règlement 02-003 full text — only the City's paraphrase in the RAAV metadata. → Service du greffe.
  • Banque d'information 311 — now VPN-restricted (403). Our taxonomy reconstruction is a proxy. → request access as a participant.
  • RAAV's 33 ARTERE=1 segments inside villes liées — found via the GPKG, not re-downloadable to confirm. Agrees with an independent 311 finding, so probably right.
  • PJCCI / CA25 / SSL as RTSS operators — one pass found them; a later sample returned only MTQ. Not disproven, not sampled.
  • The AREQ zone in Hydro-Québec's pole KMZ — blank manager, no legend. Possibly the CSEM buried network. Unconfirmed.
  • Québecor's bus-shelter contract status in 2026 — sources are the 2012–13 announcements (runs to 2032).
  • HQ open data is CC-BY-NC — a real constraint if commercialised.

7The process lesson

Three of our four worst errors share one shape: we trusted a schema, a convenient number, or a plausible column name instead of looking at the distribution.

  • The dictionary permitted MTQ → we assumed MTQ was in the data.
  • A column named ARRONDISSEMENT → we assumed it meant "where it is."
  • A value 1 in PROVENANCE_TELEPHONE → we assumed a boolean.

The fourth was the mirror image: we disbelieved a correct finding (the obfuscation rule) because our own reasoning about the coordinates looked sound. It was sound and wrong.

The one ruleLook at the values. Always. The schema is a claim, not evidence.
